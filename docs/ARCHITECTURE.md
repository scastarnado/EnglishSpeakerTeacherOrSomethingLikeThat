# Architecture Documentation

## Overview

C1 Speaking Coach is a local-first desktop application built with Electron, React, TypeScript, and Python. The architecture prioritizes privacy, offline operation, and evidence-based assessment.

## System Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                     Desktop Application (Electron)               │
```

## Component Breakdown

### 1. Desktop Application (Electron)

#### Main Process

- Application lifecycle management
- Window creation and management
- AI service spawning and monitoring
- IPC handling
- SQLite database access
- File system operations
- Audio file management

**Key Responsibilities:**

- Start/stop Python AI service on app launch/quit
- Maintain service health checks
- Handle service crashes with auto-restart
- Manage authentication tokens for local service
- Provide secure IPC channels to renderer

#### Preload Scripts

- Expose safe IPC channels to renderer
- No direct Node.js access from renderer
- Typed API surface for main-renderer communication

#### Renderer Process (React)

- User interface
- Session flow management
- Audio recording via MediaRecorder API
- Real-time UI updates
- Results visualization
- Progress tracking interface

**Technology Stack:**

- React 18+
- TypeScript (strict mode)
- Vite (build tool)
- TailwindCSS (styling)
- Radix UI (accessible components)
- Zustand (state management)
- TanStack Query (data fetching)
- React Router (navigation)
- Zod (validation)
- Recharts (data visualization)

### 2. AI Service (Python FastAPI)

Local HTTP service handling all AI inference and audio processing.

**Port:** Dynamic (chosen at startup, communicated to Electron)  
**Binding:** localhost only  
**Authentication:** Bearer token generated by Electron

#### Core Modules

```sql
CREATE TABLE assessments (

    FOREIGN KEY (turn_id) REFERENCES turns(id)

-- Audio analysis metrics
CREATE TABLE audio_metrics (
```

## Data Flow

### Session Flow: Part 1 Practice

```
1. User clicks "Start Part 1 Practice"
   └─> Frontend: Initialize session state
   └─> Main process: Create session record in DB
   └─> Main process: Verify AI service health
   └─> Main process: Test Ollama connection

2. User grants microphone permission
   └─> Renderer: Start MediaRecorder
   └─> Frontend: Display "Listening..." state

3. Examiner asks first question
   └─> Frontend: Send request to AI service via IPC
   └─> Main: Forward to AI service
   └─> AI Service: Call interlocutor agent
   └─> AI Service: Generate structured response
   └─> AI Service: Synthesize speech with TTS
   └─> Main: Save turn to database
   └─> Frontend: Play audio, display question

4. User speaks answer
   └─> Renderer: Record audio blob
   └─> Main: Save audio file
   └─> Main: Send to AI service for transcription
   └─> AI Service: Run Whisper transcription
   └─> AI Service: Extract timestamps
   └─> Main: Save turn with transcript to DB
   └─> Frontend: Display transcript

5. Examiner responds
   └─> (Repeat step 3 with conversation history)

6. Part 1 complete
   └─> Frontend: Transition to results or next part
   └─> Main: Update session status

7. Assessment generation
   └─> Main: Gather all turns, transcripts, audio metrics
   └─> Main: Send to AI service
   └─> AI Service: Run assessment engine
   └─> AI Service: Validate output schema
   └─> Main: Save assessment to DB
   └─> Frontend: Display results
```

## Security

### Electron Security

- **Context Isolation**: Enabled
- **Node Integration**: Disabled in renderer
- **Content Security Policy**: Strict CSP headers
- **IPC Allowlist**: Explicit channel registration
- **No eval()**: Disabled in all contexts

### AI Service Security

- **Localhost Only**: Bind to 127.0.0.1
- **Authentication**: Bearer token (generated on startup)
- **No External Access**: Firewall rules prevent external connections
- **Input Validation**: All inputs validated with Pydantic
- **Path Traversal**: File operations restricted to app data directory
- **Sandboxing**: Service runs with minimal privileges

### Data Security

- **No Encryption at Rest**: Not implemented (local-only, single-user)
- **Secure Deletion**: Overwrite audio files on deletion
- **Export Sanitization**: Remove sensitive metadata from exports

## Performance Optimizations

### Audio Processing

- Stream audio to service (no full upload before transcription)
- Progressive transcription where supported
- Audio file compression for storage (FLAC)
- Async transcription (non-blocking UI)

### LLM Inference

- Preload models on app start
- Maintain persistent Ollama connection
- Cache common examiner phrases
- Stream responses for long generation

### TTS

- Cache generated audio for repeated instructions
- Pregenerate common phrases on startup
- Parallel synthesis for multiple responses

### Database

- Indexes on frequently queried fields
- Batch inserts for transcript words
- Lazy load audio files
- Archive old sessions to separate DB

### Frontend

- Virtual scrolling for long transcripts
- Lazy load session history
- Code splitting by route
- Memoized expensive computations
- Web Workers for heavy client-side processing

## Error Handling

### Recovery Strategies

| Error Scenario          | Strategy                                            |
| ----------------------- | --------------------------------------------------- |
| AI service crash        | Auto-restart service, resume session                |
| Ollama timeout          | Retry once, then fail gracefully with message       |
| Invalid LLM output      | Schema repair attempt, then simplified prompt retry |
| Microphone disconnected | Pause recording, prompt user to reconnect           |
| Database locked         | Retry with exponential backoff                      |
| Out of disk space       | Warn user, offer to delete old sessions             |
| Model not found         | Prompt to download model, offer alternatives        |
| Transcription failure   | Allow manual transcript entry                       |

### State Persistence

- Autosave session state every 30 seconds
- Write-ahead logging for database
- Crash recovery on startup
- Offer to resume interrupted sessions
